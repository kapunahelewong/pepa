Setting Up the API Reference

How to connect your own OpenAPI spec and configure the interactive API explorer.

443 words·3 min read

The API Reference tab is powered by Scalar, an open-source interactive API explorer. Pepa ships with a working example spec built on JSONPlaceholder — a free hosted service with real GET, POST, PUT, PATCH, and DELETE endpoints for posts, comments, todos, and users. You can execute requests directly in the explorer with no API key. This guide explains how to swap it out for your own spec.

How it works

The pipeline has three parts:

1

Your spec lives in openapi/

Any .yaml or .json file you place in apps/docs/openapi/ is your source of truth.

2

It gets copied to public/ at build time

The copy-openapi script (run automatically before pnpm dev and pnpm build) copies everything in openapi/ to public/openapi/, where Next.js serves it as a static file.

3

Scalar loads it at runtime

The API reference page points Scalar at the served file. Scalar fetches it in the browser and renders the interactive explorer — no server required.

Swap in your own spec

1

Add your spec file

Copy your OpenAPI 3.x spec into apps/docs/openapi/. The filename is up to you:

apps/docs/openapi/
├── example-api.yaml   ← JSONPlaceholder demo (can delete this)
└── your-api.yaml      ← your real spec
💡

Both OpenAPI 3.0 and 3.1 work. JSON is also accepted alongside YAML.

Remove the example spec (optional)

Delete apps/docs/openapi/example-api.yaml once your spec is wired up. It's only there to give the tab something to render on a fresh clone.

4

Restart the dev server

The predev step runs copy-openapi automatically, so your spec is copied to public/openapi/ before the server starts. Navigate to the API Reference tab to confirm Scalar is loading your spec.

Multiple specs

If you have more than one API (for example, a public API and an internal one), you can drop multiple files into openapi/ and switch between them:

apps/docs/openapi/
├── public-api.yaml
└── internal-api.yaml

Point Scalar at whichever one you want as the default in page.tsx. To let readers switch between specs, you'd need to extend the page component to render a selector — this isn't built in yet, but the files are all served statically so client-side switching is straightforward to add.

Common issues

The explorer shows 'Failed to fetch' or a network error

The spec file isn't being served. Check that:

  1. Your file is in apps/docs/openapi/ (not apps/docs/public/openapi/ directly)
  2. You restarted the dev server after adding the file (so predev ran copy-openapi)
  3. The filename in page.tsx matches the actual filename exactly, including extension
Scalar loads but my spec has validation errors

Scalar renders even with spec errors, but some operations may not display correctly. Run your spec through the Swagger Editor or the Scalar validator to catch issues before publishing.

The API Reference tab is slow on first load

This is expected in dev — Turbopack compiles Scalar's JavaScript module on first visit (~5 seconds). After that first compile it's cached for the rest of the dev session. Production builds are pre-compiled and load instantly.
